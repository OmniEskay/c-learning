Memory Management

Why does this matter?

The Heartbleed vulnerability (CVE-2014-0160) was a buffer over-read in OpenSSL that exposed private server keys for 66% of the internet. It was a C memory management error. Every heap exploitation technique — use-after-free, heap spray, tcache poisoning — is built on misuse of malloc/free. Understanding memory management means understanding both how to write safe code and why security exploits work the way they do.

Stack vs Heap

C programs use two main memory regions for data:

VIRTUAL ADDRESS SPACE (simplified) High addresses ┌───────────────────────────────────────────┐ │ STACK │ │ ┌─────────────────────────────────────┐ │ │ │ main() frame │ │ │ │ local vars, saved registers │ │ │ └─────────────────────────────────────┘ │ │ ↓ (grows downward on most platforms) │ │ │ │ (unmapped gap) │ │ │ │ ↑ (grows upward) │ │ ┌─────────────────────────────────────┐ │ │ │ HEAP │ │ │ │ malloc/calloc/realloc memory │ │ │ └─────────────────────────────────────┘ │ ├───────────────────────────────────────────┤ │ BSS — uninitialised global/static vars │ ├───────────────────────────────────────────┤ │ Data — initialised global/static vars │ ├───────────────────────────────────────────┤ │ Text — compiled machine code │ └───────────────────────────────────────────┘ Low addresses STACK characteristics: HEAP characteristics: ✓ Automatic lifetime ✗ Manual lifetime (malloc/free) ✓ Very fast (adjust SP) ✓ Can be any size ✓ No fragmentation ✓ Lifetime crosses function calls ✗ Limited size (~8 MB) ✗ Slower (allocator overhead) ✗ Lost when fn returns ✗ Risk of leaks and fragmentation

malloc, calloc, realloc, free

#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main(void) {
    /* malloc: allocate n bytes, UNINITIALISED */
    int *a = malloc(10 * sizeof(int));
    if (!a) { perror("malloc"); return 1; }
    /* a contains garbage — must initialise before reading */

    /* calloc: allocate n elements of size, ZERO-INITIALISED */
    int *b = calloc(10, sizeof(int));
    if (!b) { free(a); return 1; }
    /* b is guaranteed to be all zeros */

    /* realloc: resize existing allocation */
    int *c = malloc(5 * sizeof(int));
    if (!c) { free(a); free(b); return 1; }
    /* ... fill c with 5 ints ... */

    int *c2 = realloc(c, 10 * sizeof(int));  /* resize to 10 ints */
    if (!c2) {
        free(c);   /* realloc failed, original c still valid */
        free(a); free(b);
        return 1;
    }
    c = c2;   /* on success, c is now invalid; use c2 (or reassign) */

    free(a);
    free(b);
    free(c);  /* c now points to realloc'd memory */
    return 0;
}

Memory Leaks

A memory leak occurs when heap memory is allocated but never freed:

/* Common leak patterns */

/* Pattern 1: early return without freeing */
int *data = malloc(100 * sizeof(int));
if (some_error_condition) {
    return -1;  /* BUG: data leaked! */
}
free(data);

/* Pattern 2: reassigning a pointer without freeing */
char *s = malloc(10);
s = malloc(20);  /* BUG: original 10 bytes leaked */

/* Pattern 3: losing the only pointer to heap memory */
void leak(void) {
    int *p = malloc(sizeof(int));
    *p = 42;
}  /* p goes out of scope — malloc'd block unreachable, leaked */

Detecting Leaks with Valgrind

Valgrind's memcheck tool instruments your program at runtime to detect heap errors:

gcc -g -o prog prog.c
valgrind --leak-check=full --show-leak-kinds=all ./prog

Sample Valgrind output for a leak:

==12345== LEAK SUMMARY:
==12345==    definitely lost: 40 bytes in 1 blocks
==12345==    indirectly lost: 0 bytes in 0 blocks
==12345== ERROR SUMMARY: 1 errors from 1 contexts

==12345== 40 bytes in 1 blocks are definitely lost in loss record
==12345==    at 0x4C2FB0F: malloc (in vgpreload_memcheck.so)
==12345==    by 0x400526: main (prog.c:8)

Alternatively, compile with AddressSanitizer (no extra tool needed):

gcc -g -fsanitize=address,leak -o prog prog.c
./prog

Segmentation Faults

A segfault (SIGSEGV) occurs when your program accesses memory it doesn't own:

/* Common segfault causes */

/* 1. NULL dereference */
int *p = NULL;
*p = 5;            /* SIGSEGV */

/* 2. Out-of-bounds array access */
int arr[3] = {1, 2, 3};
int x = arr[10];   /* reading random memory — undefined behaviour */

/* 3. Use after free */
int *q = malloc(sizeof(int));
free(q);
*q = 42;           /* SIGSEGV or silent corruption */

/* 4. Stack overflow (deep recursion) */
void infinite(void) { infinite(); }  /* eventually SIGSEGV */

To debug a segfault with GDB:

gcc -g -o prog prog.c
gdb ./prog
# In GDB:
run
bt    # print backtrace — shows which line caused the crash

Diff: Use-After-Free Bug

/* BROKEN: use-after-free */
#include <stdlib.h>
#include <stdio.h>

int main(void) {
    int *p = malloc(sizeof(int));
    if (!p) return 1;
    *p = 42;

    free(p);       /* memory returned to allocator */
    printf("%d\n", *p);  /* BUG: undefined behaviour */
    /* May print 42, 0, crash, or anything else */

    return 0;
}
/* FIXED: don't access after free; NULL the pointer */
#include <stdlib.h>
#include <stdio.h>

int main(void) {
    int *p = malloc(sizeof(int));
    if (!p) return 1;
    *p = 42;

    printf("%d\n", *p);  /* read BEFORE freeing */
    free(p);
    p = NULL;   /* guard: accessing NULL will crash immediately */
                /* making the bug obvious rather than silent */

    return 0;
}
What was wrong: After free(p), the allocator may immediately reuse that memory for another allocation. Reading or writing through p after the free is undefined behaviour — the most dangerous kind, because it often appears to work in tests but fails randomly in production.

Why the fix works: Print before freeing. After freeing, setting p = NULL means any subsequent accidental dereference will immediately crash with a null-pointer error — a loud, obvious bug — rather than silent memory corruption.
⚠ Common Mistakes
  • Double free — Calling free(p) twice corrupts the allocator's free list. This is a security vulnerability (heap exploitation). Setting p = NULL after free() makes a double-free a safe no-op since free(NULL) is defined to do nothing.
  • Forgetting to check realloc's return value — If realloc fails and you do p = realloc(p, n), p becomes NULL but the original memory is still allocated and now unreachable — a guaranteed leak. Always use a temporary variable.
  • Stack overflow from large local arrays — char buf[10000000]; on the stack will likely crash (stack is limited to ~8 MB). Large buffers should be heap-allocated with malloc.

Practice Exercises

Basic Safe string duplicate

Implement char *safe_strdup(const char *s) that allocates exactly enough memory to hold a copy of s (including the null terminator), copies it, and returns the pointer. The caller must free the result.

Show hint
Allocate strlen(s) + 1 bytes with malloc. Copy with strcpy or memcpy. Return the pointer. Return NULL if malloc fails. This is how POSIX strdup() works.
Intermediate Memory-safe vector (dynamic array)

Implement a dynamic integer array with a struct: { int *data; size_t size; size_t capacity; }. Write vec_push, vec_get, vec_free. Double capacity when full. Always check malloc/realloc results.

Show hint
Start with capacity 4. In vec_push: if size==capacity, realloc to capacity*2. Save the realloc result in a temp ptr; only update vec->data if not NULL. vec_free: free(vec->data), set data=NULL, size=capacity=0.
Challenge Find and fix all bugs with Valgrind

Write a program that intentionally contains: (1) a memory leak, (2) a use-after-free, (3) an out-of-bounds read. Compile with -g and run under Valgrind. Fix each bug based on Valgrind's output. Document what Valgrind reported for each issue.

Show hint
Use --leak-check=full for leaks. Use-after-free appears as "Invalid read of size N". Out-of-bounds: access arr[n] on an n-element array. Fix in order: easiest to hardest.

Checkpoint Quiz

Test your understanding

1. What is a memory leak?
A memory leak is heap memory that was allocated with malloc/calloc but never returned with free. It grows the process footprint indefinitely.
2. calloc differs from malloc in that it:
calloc(n, size) allocates n*size bytes and zero-initialises them. malloc leaves memory uninitialised — reading uninitialised memory is undefined behaviour.
3. What tool detects memory leaks and invalid accesses at runtime?
Valgrind (memcheck tool) instruments the program to track all heap allocations and detects leaks, use-after-free, and invalid reads/writes.
