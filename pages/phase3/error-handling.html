Error Handling

Why does this matter?

This is exactly how system calls signal failure in Unix. Every time your program opens a file, connects to a network socket, or allocates memory, the kernel returns a value indicating success or failure and sets errno to explain why. Production C code that ignores errors becomes a mystery when it fails in the field — often silently corrupting data or hanging indefinitely. Good error handling is what makes the difference between a toy program and production-grade software.

errno: The Global Error Code

errno is a global integer variable (in <errno.h>) that is set by the C library when a system call or library function fails. It is only meaningful immediately after a failure — a successful call may or may not reset it.

#include <stdio.h>
#include <errno.h>
#include <string.h>

int main(void) {
    FILE *fp = fopen("nonexistent.txt", "r");
    if (fp == NULL) {
        /* errno is now set — check it immediately */
        printf("errno = %d\n", errno);                /* e.g., 2 */
        printf("Meaning: %s\n", strerror(errno));     /* "No such file or directory" */
    }

    /* Another common pattern: reset errno before the call */
    errno = 0;
    long val = strtol("abc", NULL, 10);
    if (errno != 0) {
        perror("strtol");
    }

    return 0;
}

perror and strerror

#include <stdio.h>
#include <errno.h>
#include <string.h>

void demo_errors(void) {
    FILE *fp = fopen("/root/secret", "r");   /* Will fail with EACCES if not root */
    if (!fp) {
        /* perror: prints "prefix: error message\n" to stderr */
        perror("fopen");
        /* Output: "fopen: Permission denied" */

        /* strerror: returns the error string for any errno value */
        fprintf(stderr, "Error %d: %s\n", errno, strerror(errno));

        return;
    }
    fclose(fp);
}

Return-Code Conventions

C functions communicate errors through their return value. There is no standard convention — each function documents its own:

Function typeSuccessFailure
fopen, mallocnon-NULL pointerNULL (sets errno)
fread, fwritenumber of items processedfewer items + feof/ferror
most POSIX syscalls0 or non-negative-1 (sets errno)
strcmp, memcmp0 = equalnon-zero (not an error)
Custom functions0 or valid pointernegative / NULL (document it!)

Propagating Errors

Production C code propagates errors up the call stack rather than calling exit() immediately:

#include <stdio.h>
#include <stdlib.h>
#include <errno.h>

/* Returns 0 on success, -1 on error */
int load_config(const char *path, char *buf, int buflen) {
    FILE *fp = fopen(path, "r");
    if (!fp) return -1;   /* errno already set by fopen */

    if (!fgets(buf, buflen, fp)) {
        int saved = errno;   /* save before fclose might change it */
        fclose(fp);
        errno = saved;
        return -1;
    }

    fclose(fp);
    return 0;
}

int main(void) {
    char config[256];
    if (load_config("config.txt", config, sizeof(config)) != 0) {
        perror("load_config");   /* prints the preserved errno */
        return 1;
    }
    printf("Config: %s", config);
    return 0;
}

assert for Programming Errors

#include <assert.h>

/* Use assert for conditions that should NEVER be false
   (programming errors, not user errors) */
void process(int *data, int n) {
    assert(data != NULL);      /* programmer contract: must not pass NULL */
    assert(n > 0);             /* must have at least one element */

    /* If these assertions fire in debug mode, it indicates
       a bug in the caller, not a runtime error to handle */
}

/* Assertions are disabled with -DNDEBUG (release builds)
   DO NOT use assert for user input validation or file errors */

Diff: Ignoring Error Codes

/* BROKEN: silently ignoring errors */
#include <stdio.h>

void save_data(const char *path, int *data, int n) {
    FILE *fp = fopen(path, "w");
    /* No NULL check — if fopen fails, fp is NULL */

    for (int i = 0; i < n; i++) {
        fprintf(fp, "%d\n", data[i]);
        /* If disk is full, fprintf silently fails */
    }

    fclose(fp);
    /* No check if write actually succeeded */
}
/* FIXED: check and report every failure */
#include <stdio.h>
#include <errno.h>

int save_data(const char *path, int *data, int n) {
    FILE *fp = fopen(path, "w");
    if (!fp) { perror("fopen"); return -1; }

    for (int i = 0; i < n; i++) {
        if (fprintf(fp, "%d\n", data[i]) < 0) {
            perror("fprintf");
            fclose(fp);
            return -1;
        }
    }

    if (fclose(fp) != 0) {   /* fclose can fail! (flush error) */
        perror("fclose");
        return -1;
    }
    return 0;
}
What was wrong: Ignoring return values means silent data corruption. A full disk causes fprintf to fail silently. The file may be partially written, but the caller has no way to know. On a production server, this is how databases lose data.

Why the fix works: Each operation's return value is checked. On failure, perror prints the reason and the function returns -1. Note that fclose() itself can fail — it flushes any buffered writes and must be checked.
⚠ Common Mistakes
  • Reading errno after a successful call — errno is only valid immediately after a failure. A subsequent successful call may change or clear errno. Check it before calling anything else.
  • Using errno in multi-threaded code — In threaded programs, errno is thread-local (each thread has its own copy) in POSIX. This is handled automatically by the C library.
  • Calling exit() deep in library code — Library functions should return error codes to the caller; they shouldn't exit() unilaterally. The caller must decide whether the error is fatal.

Practice Exercises

Basic Wrap a system call

Write a function FILE *safe_fopen(const char *path, const char *mode) that wraps fopen, prints a descriptive error message to stderr on failure (including filename and errno string), and returns NULL on failure.

Show hint
fprintf(stderr, "Cannot open '%s': %s\n", path, strerror(errno)). Return NULL. On success, just return the FILE*.
Intermediate Error-safe numeric parsing

Write int parse_int(const char *s, long *out) that uses strtol to safely parse an integer string. Return 0 on success, -1 on error (non-numeric input, overflow, empty string). Set *out only on success.

Show hint
Set errno=0, call strtol with &endptr. Check: errno != 0 (overflow), endptr == s (no digits), *endptr != '\0' (trailing garbage). All three indicate failure.
Challenge Build an error-handling library

Design a simple error-handling system: an Error struct with a code, message, and file/line info. Write macros RETURN_ERR(code, msg) and CHECK(expr). Implement a function chain that propagates errors with context through 3 levels of calls.

Show hint
Use the __FILE__ and __LINE__ macros in error macros. The Error struct: {int code; char msg[128]; const char *file; int line;}. RETURN_ERR fills the struct and returns. CHECK calls a function and returns its error if non-zero.

Checkpoint Quiz

Test your understanding

1. What global variable is set when a system call fails?
errno is defined in <errno.h> and set by the C library on failure. It's thread-local in modern implementations.
2. Which function prints a human-readable error message for the current errno?
perror(prefix) prints "prefix: error string\n" to stderr. strerror(errno) returns the error string for use in your own printf.
3. By convention, a C function returns what value to signal failure?
Conventions vary: integer functions often return -1 for failure; pointer functions return NULL. Always read the man page for each function's specific convention.
